# `ymerflow-gcp` plugin — GKE cluster type + GCS storage type

## Goal & scope

Add a new backend+frontend plugin, `plugins/ymerflow-gcp`, that plugs into the existing
`cluster_provider_handlers` / `storage_protocol_handlers` extension points
(`docs/plans/cluster-admin-ui.md`, `docs/plans/done/short-lived-storage-credentials-*.md`) to add:

- a `gke` `Cluster.cluster_type` — the plugin **provisions a real GKE cluster** via the GCP APIs
  (not just "connect to one that already exists" — see Design decisions), and
- a `gcs` `StorageBackend.protocol` — the plugin **replaces core's `gcs` stub** with a real
  implementation (bucket + service-account provisioning, `static-key` credential strategy).

Both use a single **Google OAuth consent flow** run by the admin to obtain the credentials the
plugin then uses to create whatever service accounts, GKE clusters, and buckets are needed —
explicitly *not* "paste a service-account JSON key you generated by hand," to avoid the same
manual-provisioning friction the minikube self-service flow was built to eliminate
(`docs/plans/done/remote-cluster-provisioning-and-registry.md`,
`docs/plans/minikube-cluster-registration-ux.md`).

This plan follows the SDK's plugin-authoring model
(`deps/Ymerflow-plugin-sdk/docs/{authoring,backend-hooks,frontend-hooks,distributing}.md`) and the
existing `plugins/billing` package as a structural template.

## Depends on

[cleanup-cluster-connect.md](cleanup-cluster-connect.md) — `ClusterProvider.connect()` must return
a `K8sClient` before `GkeClusterProvider` can be written (it needs a `K8sClient` subclass to install
its token-refresh logic on; see Design decisions below). Implement that plan first.

## Background — current state

- **`ClusterProvider` / `StorageProtocolHandler`** (`backend/services/cluster_providers/__init__.py`,
  `backend/services/storage_protocols/__init__.py`) are fan-out-hook registries. Core registers its
  own built-in providers through the *same* `nagelfluh.hooks` `cluster_provider_handlers` /
  `storage_protocol_handlers` entry points a plugin would use — "core has no special precedence
  over plugins" is stated explicitly in both files' docstrings. A duplicate name registered twice
  raises `ValueError` at first use (`get_cluster_provider`/`get_protocol_handler`).
- **Core already has an unimplemented `gcs` stub**: `backend/services/storage_protocols/gcs.py`
  (`GcsProtocolHandler`, every method raises `NotImplementedError`), registered in root `setup.py`,
  plus a matching frontend placeholder `frontend/src/storageProviders/GcsStorageForm.jsx`
  registered via `registerHook('storage_protocol_forms', ...)` in `frontend/src/App.jsx`. Both
  their own docstrings frame real GCS support as deferred/future work. **This plan removes both**
  (backend stub + `setup.py` entry, frontend form + `App.jsx` registration) so the plugin can
  register the real `gcs` handler under the same name — confirmed with the user rather than
  inventing a second discriminator (`gcp-gcs` etc.), which would make `StorageBackend.protocol`
  stop meaning "which cloud API" and leave the stub UI dangling and misleading.
- **`docs/plans/cluster-admin-ui.md`** explicitly deferred a GKE provider to "a separate plan,
  added later as its own provider registration... with no changes needed to the registry/dispatch
  mechanism" — this is that plan, for the cluster side.
- **`Cluster` schema** (`backend/models/cluster.py`): `cluster_type`, `provider_config` (opaque
  JSON), `provisioning_status` (`active`/`pending`/`failed`), `registration_token_hash`. The
  `pending` self-service flow (minikube) is: admin runs a script on an *already-provisioned* host,
  the script calls back to `POST /admin/clusters/register-callback` with a kubeconfig, a `pending`
  Cluster row is created/updated, the admin's still-open dialog polls
  `GET /admin/clusters/by-registration-token` to discover it, then the admin clicks Save to
  activate (`backend/routers/admin.py:23-284`). GKE provisioning is different in *what* triggers
  the wait (the backend calling GCP APIs itself, not an external script), but the same
  `provisioning_status` field and poll-until-discovered UX shape is reusable (see Design decisions).
- **`K8sClientRegistry.get()`** (`backend/services/k8s_client.py:392-408`) calls
  `provider.connect(...)` once per cluster and caches the resulting `K8sClient` forever, for the
  life of the backend process. A GKE access token expires in ~1 hour, so `GkeClusterProvider`
  needs its `K8sClient` to self-refresh — this is exactly what
  [cleanup-cluster-connect.md](cleanup-cluster-connect.md) enables (`connect()` returning a
  `K8sClient` lets a provider subclass it).
- **Auth is Bearer-JWT, not cookie-based** (`backend/services/auth_service.py` uses `HTTPBearer`).
  This matters for the OAuth callback: Google's redirect is a plain top-level browser navigation
  with no `Authorization` header, so the callback route can't identify "which admin started this"
  via the normal `get_current_user` dependency. Needs a server-side pending-state record keyed by
  the `state` query param (see Architecture below), not a bearer-token-protected route.
- **No encryption-at-rest layer exists for secrets today** — `provider_config`/`config`/
  `registry_auth` are plain JSON/String columns; the only protection is that admin list/get
  responses never echo them back (`has_provider_config`-style booleans instead). This plan follows
  the same precedent for OAuth refresh tokens rather than introducing new crypto infrastructure.
- **`plugins/billing`** is the template for a backend+frontend plugin package: `setup.py` builds
  the frontend at `pip install` time (`ymerflow_plugin_build`), registers hooks under
  `nagelfluh.hooks`, and ships its own Alembic branch under `nagelfluh.migration_dirs`
  (`billing/migrations/`). `ymerflow-gcp` follows the identical shape.

## Design decisions already settled (confirmed with the user)

1. **`gcs` naming**: core's stub is removed; the plugin owns `gcs` for real.
2. **GKE scope**: full lifecycle creation. The plugin creates the actual GKE cluster via the GCP
   Container API, rather than asking the admin to create one out-of-band and paste connection
   details. Rationale (from the user): pointing at an already-existing cluster reproduces the same
   "provisioning and auth problems for the user" the minikube self-service flow exists to avoid.
   Deletion is explicitly out of scope (settled decision E below) — the plugin never tears down
   GCP infrastructure it created.
3. **Credential model**: a **Google OAuth consent screen loop** run by the admin (not a pasted
   service-account JSON key, not reliance on the backend's own ambient ADC). The resulting
   credential is then used server-side to create whatever service accounts / IAM bindings are
   needed to (a) manage a GKE cluster and (b) create/access GCS buckets.
4. **GCS credential strategy**: `static-key` only for v1 (matches `MinioProtocolHandler`'s existing
   shape — the only strategy fully wired end-to-end today). `short-lived` (IAM Credentials API
   impersonation) is future work, same interface.

## Design decisions — round 2 (settled)

Resolved in discussion with the user. No persisted GCP connection resource, no CRUD surface, no
new DB tables/migrations for this plugin at all — the OAuth credential is used once, transiently,
inside the add/edit dialog, purely to *mint* narrowly-scoped service accounts, and is then
discarded. This is simpler than the round-1 draft above (kept for context on the trade-offs
considered) in every dimension: A/B collapse into "inline, per-dialog, forgotten after use," C
resolves cleanly to a self-refreshing service-account credential (no core token-refresh plumbing
needed for the *admin OAuth token* — only, still, for the *stored SA credential*, see below), and
E disappears entirely (no GCP-side deletion logic to write at all).

### A + B. Per-dialog, transient OAuth — no shared connection resource

The "Connect Google Cloud" step lives **inside** the Add/Edit Cluster dialog (type `gke`) and the
Add/Edit Storage Backend dialog (protocol `gcs`), not as a separate admin resource. Admin clicks
"Connect Google Cloud" in the dialog → OAuth popup → backend caches the resulting access/refresh
token **in-process, in memory, keyed by an opaque session id**, scoped to that one open dialog.
The admin fills in the rest of the type-specific fields (project id, region, cluster name /
bucket name) and clicks Save; the session id rides along in that one request. The backend uses
the cached OAuth credential *only* to provision the purpose-built resources (see C), then deletes
it from the cache — success or failure — so nothing OAuth-derived is ever written to the
database. (Confirmed safe to hold this cache in a plain in-process dict: `backend/run.sh` and
`backend/Dockerfile`'s `CMD` both run a single uvicorn worker, no `--workers` flag, so there's
exactly one process to hold it.)

### C. What actually gets stored, and how GKE credential refresh is solved

The OAuth admin token itself is never persisted (per A/B) — so **it can't be what `connect()`
uses for ongoing cluster access anyway**. Instead, the transient OAuth-admin-authenticated call
provisions a **purpose-built service account**, and *its* key is what gets stored in
`provider_config`/`config` and used forever after:

- **`gke`**: after the cluster exists, create a dedicated service account narrowly scoped for
  ongoing Kubernetes access to *this* cluster (IAM role granting `container.clusters.get` +
  the ability to authenticate to the cluster's API server — `roles/container.developer` is the
  closest built-in role; a tighter custom role scoped via an IAM Condition to the specific cluster
  resource is worth evaluating during implementation), create a key for it
  (`serviceAccounts.keys.create`), and — since GKE additionally requires a Kubernetes-level RBAC
  grant, not just the IAM role, for a principal to actually act inside the cluster (the same gap
  `k8s/rbac/backend-jobs-rbac.yaml` closes for the in-cluster case) — apply an equivalent
  ClusterRoleBinding for this SA's identity using the *admin* OAuth credential (which, being a
  real user login, already has full `cluster-admin`-equivalent access to a cluster it just
  created) before that admin credential is discarded. `provider_config` ends up holding
  `{project_id, region, cluster_name, sa_key}` — no OAuth token anywhere in it.
- **`gcs`**: create a bucket + a service account scoped *only* to that bucket
  (`roles/storage.objectAdmin` bound at the bucket level via the bucket's own IAM policy, not the
  project's), create a key for it, store `{sa_key}` (plus bucket/project identifiers) as the
  `static-key` credential — mirrors `MinioProtocolHandler`'s existing "create scoped admin
  identity, store its key" shape exactly.

The stored SA key mints access tokens that expire hourly.
[cleanup-cluster-connect.md](cleanup-cluster-connect.md) makes `ClusterProvider.connect()` return
a `K8sClient` instead of raw kubeconfig data. `GkeClusterProvider.connect()` returns a
`GkeK8sClient(K8sClient)` subclass that overrides `_ensure_initialized()` to build the initial
connection from the stored SA key and install a `refresh_api_key_hook` (a `kubernetes_asyncio`
`Configuration` feature, called before every request — see that plan) backed by
`google.auth`'s self-refreshing `service_account.Credentials`. No changes to `K8sClient` or
`ClusterProvider` beyond what that plan already makes — this part is pure plugin code.

### D. Async GKE provisioning — confirmed

GKE cluster creation is a long-running GCP `Operation` (minutes). The Cluster row is created
immediately with `provisioning_status="pending"`, an `asyncio` background task (kicked off from
the create route — network-bound polling, not compute, so it doesn't run afoul of CLAUDE.md's "no
expensive backend operations" rule) drives the Container API operation to completion, then does
the SA-provisioning + RBAC-binding step from C, then flips the row to `active` (after running
`test_connection`) or `failed`. `GkeClusterForm` polls the existing `GET /admin/clusters` list (or
a new per-id status endpoint, cheaper than refetching the whole list) the same way
`MinikubeClusterForm` polls `by-registration-token` — just against an id it already has instead of
a client-generated token. The transient OAuth session (from A/B) needs a TTL long enough to
survive this whole task for `gke` (recommend ~30 min; `gcs` provisioning is synchronous and fast,
so a couple of minutes is enough there). A backend restart mid-provisioning loses the in-flight
task and leaves the row stuck in `pending` with no OAuth session left to resume it — acceptable
for v1 (the admin deletes/retries), not solved here.

### E. Teardown / retirement — no GCP-side deletion, at all

Confirmed out of scope. "Retire" (`active=false`) on a `gke` Cluster or a `gcs` StorageBackend
never calls any GCP delete API — it's purely the existing in-app soft-retire behavior, unchanged
from every other provider. Deleting the underlying GCP cluster/bucket/service-accounts is a manual
admin action outside this plugin (`gcloud`/console). This removes an entire category of
destructive-action design work from this plan.

### F. IAM scope

The transient admin OAuth grant requests the broad `https://www.googleapis.com/auth/cloud-platform`
scope (needed to create clusters/SAs/buckets/IAM bindings in the first place). The **service
accounts the plugin creates and persists**, by contrast, get narrow, purpose-scoped roles only:
the `gke` SA gets cluster-scoped Kubernetes access (see C), the `gcs` SA gets bucket-scoped
storage admin access (see C) — neither gets `cloud-platform`, and neither gets project-wide
permissions beyond what's operationally necessary.

## Architecture

### New plugin package: `plugins/ymerflow-gcp/`

```
plugins/ymerflow-gcp/
  setup.py                          # mirrors plugins/billing/setup.py — no migrations/models needed
  pyproject.toml                    # [build-system].requires: ymerflow-plugin-build (SDK, build-only)
  gcp_plugin/
    __init__.py                     # hook entry points
    oauth.py                        # authorization-url / code-exchange + the transient in-memory session cache
    router.py                       # /admin/gcp/oauth/start, /admin/gcp/oauth/callback
    cluster_provider.py             # GkeClusterProvider(ClusterProvider) + GkeTokenSource
    storage_protocol.py             # GcsProtocolHandler(StorageProtocolHandler)
    provisioning.py                 # asyncio background task: create/poll GKE Operation, SA + RBAC setup
  frontend/
    package.json                    # nagelfluh.remoteName: "ymerflow_gcp"
    src/
      index.jsx                     # registerHook('cluster_provider_forms', ...) / ('storage_protocol_forms', ...)
      GcpConnectButton.jsx          # shared "Connect Google Cloud" component, used inline by both forms
      GkeClusterForm.jsx
      GcsStorageForm.jsx
```

No new DB models/migrations — `provider_config`/`config` (already-existing opaque JSON columns)
are all the persistence this plugin needs; the OAuth session itself never touches the database
(see A/B/D above).

### OAuth flow (addresses the Bearer-JWT-not-cookie constraint in Background)

1. Admin (already authenticated to the SPA via Bearer JWT), inside the `GkeClusterForm`/
   `GcsStorageForm` dialog, clicks "Connect Google Cloud." Frontend calls
   `POST /admin/gcp/oauth/start` (bearer-authenticated, `require_admin`), which generates a random
   opaque `session_id`, stores a placeholder entry in the in-process dict keyed by it (TTL ~30 min
   for `gke`, a few min for `gcs` — see D), and returns both `session_id` and Google's
   authorization URL (`accounts.google.com/o/oauth2/v2/auth`, `scope=cloud-platform`,
   `access_type=offline`, `prompt=consent`, `state=session_id`).
2. Frontend opens that URL in a popup (plain browser navigation — no custom headers possible,
   which is exactly why `state` carries the pre-registered `session_id` instead of relying on
   `get_current_user` at callback time).
3. Google redirects to `GET /admin/gcp/oauth/callback?code=...&state=...` (unauthenticated route —
   correlated purely via the pre-registered `session_id`/`state`, not a header). Backend exchanges
   `code` for `{access_token, refresh_token}`, stores it in the same in-memory entry (replacing the
   placeholder), and serves a minimal HTML page that `postMessage`s `{ok: true, session_id}` back
   to the opener and closes the popup.
4. `GcpConnectButton` (rendered inline inside each dialog, not a separate persisted resource)
   receives that message and marks the dialog "Connected," keeping `session_id` in the dialog's own
   local component state.
5. The admin fills in the rest of the type-specific fields and clicks Save. The Create/PATCH
   request body includes `session_id` alongside `cluster_type`/`provider_config` (or
   `protocol`/`config`) purely as a transient handle — never persisted as part of either.
6. The admin route looks up the cached OAuth credential by `session_id`, uses it *only* to drive
   provisioning (C), and deletes the cache entry once provisioning finishes — success or failure.

### `GkeClusterProvider` (backend/`cluster_provider.py`) — needs cleanup-cluster-connect.md

- `provider_config = {"project_id", "region", "cluster_name", "node_count", "machine_type", "sa_key"}` (exact field set finalized during implementation).
- Provisioning (triggered from the admin create route via the background task in D, using the
  transient OAuth session's credential, not `connect()`): `google-api-python-client`'s `container`
  v1 API (`projects.locations.clusters.create`), poll the returned `Operation`; then create the
  narrowly-scoped `gke`-access SA + key and apply the ClusterRoleBinding (both still using the
  same transient OAuth session, per C) before discarding it.
- `connect(provider_config, namespace)`: returns a `GkeK8sClient(K8sClient)` instance (not built
  yet — deferred `_ensure_initialized()`, same as base `K8sClient`) constructed from the stored
  `sa_key`. `GkeK8sClient._ensure_initialized()` builds the kubeconfig from `sa_key`, calls
  `super()._ensure_initialized()`, then installs a `refresh_api_key_hook` on
  `self.batch_api.api_client.configuration` / `self.core_api.api_client.configuration` (per C).
- `test_connection`: reuses `ClusterProvider`'s default (list-namespaces) once connected.

### `GcsProtocolHandler` (backend/`storage_protocol.py`)

- `provision(project, backend)`: using the transient OAuth session's credential (passed in from
  the admin route, same as `gke`), create a bucket, a service account scoped to just that bucket
  (`roles/storage.objectAdmin` bound on the bucket's own IAM policy, not the project's), and a key
  for it — this key is the credential persisted for `static-key` use, mirroring
  `MinioProtocolHandler`'s existing shape (create scoped admin identity, store its key). Fast
  enough to stay synchronous (no async-provisioning-task needed here, unlike `gke`).
- `mint()`: `NotImplementedError` for v1 (static-key only, per settled decision 4).
- `test_connection`: list the bucket (`storage.buckets.get` or a cheap `objects.list` with
  `maxResults=1`) using the stored SA key, timeout-bounded like `ClusterProvider`'s default.

## Implementation order

1. Implement [cleanup-cluster-connect.md](cleanup-cluster-connect.md) first (separate plan).
   Then, core removal (this plan only): delete the `gcs` stub —
   `backend/services/storage_protocols/gcs.py`, its `setup.py` entry,
   `frontend/src/storageProviders/GcsStorageForm.jsx`, and its `App.jsx` registration.
2. Plugin skeleton (`setup.py`, `pyproject.toml`, empty hook stubs, no models/migrations needed) —
   confirm it installs and registers with zero behavior (`pip install -e plugins/ymerflow-gcp`,
   restart backend, confirm `gke`/`gcs` show up as *empty-but-present* provider types with no
   crash).
3. OAuth router (`start`/`callback` + the in-memory session cache) — testable standalone via
   `curl`/`/docs` before either provider consumes it, using the already-authenticated `gcloud`
   session in this environment as the human test path (Google Cloud project `united-column-273409`
   per `gcloud config list`, or whichever test project you want this exercised against — note
   `gcloud projects list` only shows "My First Project" under this login today, so the test
   project may need to be created/selected explicitly before end-to-end testing).
4. `GcsProtocolHandler` (provision/test_connection) — the smaller, synchronous half. End-to-end
   test: create a project, confirm a real bucket + narrowly-scoped SA + key are provisioned in the
   test GCP project, confirm upload/download works through it.
5. `GkeClusterProvider` + async provisioning task (cluster creation, SA creation, RBAC binding) —
   the larger half. End-to-end test: create a `gke` Cluster via the admin UI, watch it go
   `pending` → `active`, confirm a real process can be dispatched to and run on the resulting
   cluster, confirm the token-refresh path actually gets exercised (e.g. by leaving it idle past
   an hour and confirming a job still launches).
6. Frontend forms (`GkeClusterForm`, `GcsStorageForm`, `GcpConnectButton`) wired into
   `ClustersAdminPanel`/the storage admin panel via the existing hook-driven `<select>` +
   per-type-`Component` pattern.
7. Move this plan to `docs/plans/done/`.

## Open questions not yet needing an answer to start, but flagged for later

- Exact IAM role granularity for the `gke`-access SA — a built-in `roles/container.developer`
  (project-scoped) vs. a tighter custom role bound via an IAM Condition to just the one cluster
  resource. Worth investigating during implementation; either works functionally.
- Multi-region/zone defaults, machine type, and node count for a v1 GKE cluster — sensible
  hardcoded defaults (e.g. `e2-standard-4`, 1–3 nodes, zonal not regional to minimize cost) vs.
  admin-configurable fields in the form. Leaning "admin-configurable with sensible defaults."
- Whether Kueue/registry-trust setup (the parts `dev/setup-minikube.sh` does by hand for minikube
  — see `docs/plans/done/remote-cluster-provisioning-and-registry.md`) needs an equivalent
  post-creation step for a fresh GKE cluster before it's usable for real jobs, beyond the RBAC
  binding already folded into provisioning above (GKE has its own container registry story via
  Artifact Registry, which likely sidesteps the self-signed-TLS trust problem that motivated the
  minikube registry setup — needs confirming, not assumed here).
- Backend-restart-mid-provisioning recovery for a `gke` cluster stuck in `pending` (D) — v1 leaves
  this to "admin retries," no resume-on-startup scan.
- Billing/cost visibility: this plan creates real, billed GCP infrastructure from the admin UI
  with no cost estimate shown before creation. Worth at least a "this will incur GCP charges"
  notice in `GkeClusterForm`; a real cost estimate is out of scope for v1.
